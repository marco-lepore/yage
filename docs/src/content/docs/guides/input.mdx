The `@yagejs/input` package provides a unified input layer that abstracts
keyboard, mouse, and gamepad devices behind named **actions**. You define
actions once and query them at runtime without worrying about which physical
key was pressed.

## InputPlugin Setup

```typescript
import { InputPlugin } from "@yagejs/input";

engine.use(
  new InputPlugin({
    actions: {
      jump: ["Space", "GamepadA"],
      left: ["KeyA", "ArrowLeft", "GamepadDPadLeft"],
      right: ["KeyD", "ArrowRight", "GamepadDPadRight"],
      up: ["KeyW", "ArrowUp"],
      down: ["KeyS", "ArrowDown"],
      attack: ["KeyJ", "MouseLeft", "GamepadX"],
      interact: ["KeyE"],
    },
    groups: {
      movement: ["jump", "left", "right", "up", "down"],
      combat: ["attack"],
      gameplay: ["interact"],
    },
  }),
);
```

The action map is a record of action name â†’ array of physical input codes.
Codes can be `KeyboardEvent.code` values (e.g. `"KeyA"`, `"Space"`),
synthetic mouse codes (`"MouseLeft"`, `"MouseMiddle"`, `"MouseRight"`), or
gamepad codes (see [Gamepad](#gamepad) below).

## Querying Input

Resolve the `InputManager` from the engine context and query actions by name:

```typescript
import { InputManagerKey } from "@yagejs/input";

const input = engine.resolve(InputManagerKey);
```

### Press State

```typescript
input.isPressed("jump");       // true while the key is held
input.isJustPressed("jump");   // true only on the frame the key went down
input.isJustReleased("jump");  // true only on the frame the key was released
```

### Axes and Vectors

For directional movement, turn two opposing actions into a single value:

```typescript
const h = input.getAxis("left", "right"); // -1, 0, or 1
const v = input.getAxis("up", "down");

// Or get both axes as a Vec2 in one call:
const dir = input.getVector("left", "right", "up", "down");
entity.transform.translate(dir.x * speed * dt, dir.y * speed * dt);
```

### Hold Duration

Check how long an action has been continuously held:

```typescript
const ms = input.getHoldDuration("attack");

// Convenience check â€” true if held for at least the given duration
if (input.isHeldFor("attack", 500)) {
  chargeAttack();
}
```

## Pointer (Mouse / Touch)

```typescript
// World-space position (automatically accounts for camera offset and zoom)
const pos = input.getPointerPosition();

// Raw button state
if (input.isPointerDown()) {
  shootAt(pos);
}
```

`getPointerPosition()` returns world coordinates after wiring the camera with
`input.setCamera(camera)`. If no camera is set, screen coordinates are
returned.

## Gamepad

Gamepads are polled each frame from `navigator.getGamepads()` and routed
through the same key pipeline as keyboard and mouse â€” so `isPressed`,
`isJustPressed`, hold-duration, and `listenForNextKey` all work uniformly
across devices.

### Standard-mapping codes

| Code | Description |
|------|-------------|
| `GamepadA`, `GamepadB`, `GamepadX`, `GamepadY` | Face buttons (Xbox naming) |
| `GamepadLB`, `GamepadRB` | Shoulder bumpers |
| `GamepadLT`, `GamepadRT` | Trigger buttons (fire when analog value crosses `triggerThreshold`) |
| `GamepadSelect`, `GamepadStart` | Select / Start |
| `GamepadLeftStick`, `GamepadRightStick` | Stick clicks (L3 / R3) |
| `GamepadDPadUp/Down/Left/Right` | D-pad |
| `GamepadHome` | Home / Guide button |
| `GamepadButton{0..15}` | Numeric fallback for non-standard mappings |

PlayStation users see âś•/â—‹/â–ł/â–ˇ on hardware that maps to `GamepadA/B/X/Y`
in code â€” code names are layout-canonical (Xbox), not glyph-specific.

### Analog sticks and triggers

```typescript
const move = input.getStick("left");      // Vec2, deadzoned, magnitude â‰¤ 1
const aim = input.getStick("right");
const charge = input.getTrigger("right"); // 0..1
```

Default deadzones: 0.15 for sticks (radial), 0.05 for triggers. Override per
plugin:

```typescript
new InputPlugin({
  deadzones: { stick: 0.2, trigger: 0.1 },
  triggerThreshold: 0.6,
});
```

### Multi-input movement

The platformer example accepts both keyboard and stick input on the same
action â€” let analog stick win when it has signal, fall back to digital
otherwise:

```typescript
const stickX = input.getStick("left").x;
const dx = stickX !== 0 ? stickX : input.getAxis("left", "right");
```

### Connect / disconnect events

```typescript
const dispose = input.onGamepadConnected((info) => {
  // Replays currently-known pads on subscribe â€” you don't need a separate
  // gamepads() call to discover existing controllers.
  showPlayerPrompt(`Pad ${info.index}: ${info.id}`);
});

input.onGamepadDisconnected((info) => pauseGame());

// Synchronous query â€” useful when you need ground truth right now.
const pads = input.gamepads(); // { index, id }[]
```

:::caution[Browser quirk]
Browsers gate the `gamepadconnected` event behind a first user input on the
controller (security mitigation). A pad plugged in but never used won't
trigger your `onGamepadConnected` callback until the user presses a button.
For "Press any button to start" prompts, that delay is exactly what you want;
for "show me what pads are plugged in right now", call `gamepads()`.
:::

### Press-any-button recipe

```typescript
const dispose = input.onGamepadConnected((info) => {
  showHud(`Player ready! (${info.id})`);
  dispose();
});
```

Replay-on-subscribe means this also fires for pads that were already known â€”
you handle "current and future" pads with one callback.

### Synthetic injection (testing)

```typescript
input.fireGamepadButton("GamepadA", true);  // routes through the real path
input.fireGamepadAxis("leftX", 0.7);
```

For deterministic inspector probes with a real controller plugged in, set
`pollGamepads: false` so polling can't clobber the injected state â€” pair
this with `DebugPlugin`'s `deterministicSeed` for fully deterministic test
runs:

```typescript
new InputPlugin({ pollGamepads: false, /* ... */ });
new DebugPlugin({ deterministicSeed: 42 });
```

## Rebinding Actions

Allow players to remap controls at runtime â€” works uniformly for keyboard,
mouse, and gamepad codes:

```typescript
const result = input.rebind("jump", "GamepadY", { conflict: "replace" });
if (!result.ok) {
  console.warn("Rebind conflict with", result.conflict?.action);
}
```

The `conflict` option determines what happens when the new code is already
bound to a different action *in the same group*:

- `"replace"` â€” the other action loses the binding
- `"keep-both"` â€” the new binding is added without unbinding the conflict
- `"reject"` (default) â€” the rebind is cancelled and `ok` is `false`

## Listen for Next Input

Capture the next physical input â€” for "Press a key to bind" UI:

```typescript
const code = await input.listenForNextKey();
// code is "KeyZ" | "MouseLeft" | "GamepadA" | ... | null (cancelled)
```

This intercepts the key (it does not register as a press) so the user
mashing a button to set a binding doesn't accidentally trigger the action
they're trying to rebind.

## Action Groups

Disable entire groups of actions when they shouldn't be active â€” for example,
disabling movement during a cutscene:

```typescript
input.disableGroup("movement");
input.disableGroup("combat");

// Re-enable when the cutscene ends
input.enableGroup("movement");
input.enableGroup("combat");
```

Disabled actions always return `false` / `0` when queried.

## Display Names

When building a key-bindings UI, use `getKeyDisplayName()` to convert internal
key codes into human-readable labels:

```typescript
import { getKeyDisplayName } from "@yagejs/input";

getKeyDisplayName("Space");           // "Space"
getKeyDisplayName("KeyA");            // "A"
getKeyDisplayName("ArrowLeft");       // "Left"
getKeyDisplayName("MouseLeft");       // "Left Click"
getKeyDisplayName("GamepadA");        // "A"
getKeyDisplayName("GamepadDPadUp");   // "D-Pad Up"
```
